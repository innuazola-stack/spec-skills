# PRD Writing Workflow

## Stage 1: Source Grounding

Read the user's brief, supplied research, existing product documents, tickets, designs, analytics, user feedback, and technical notes. Identify the product objective, audience, release context, and missing facts.

## Stage 2: PRD Shape Selection

Choose a structure that fits the document purpose:

- Discovery or early concept: emphasize problem, users, outcomes, hypotheses, and open questions.
- Feature delivery: emphasize scope, user flows, functional requirements, acceptance criteria, dependencies, analytics, rollout, and risks.
- Platform or API: emphasize actors, contracts, data, permissions, reliability, observability, migration, and compatibility.
- Regulated or high-risk product: emphasize traceability, compliance, guardrails, review gates, rollback, and audit evidence.

## Stage 3: Final PRD Composition

Write only final conclusions. Place the revision table at the very beginning. Use formal written Simplified Chinese. Keep sections directly tied to the PRD purpose.

## Stage 4: Quality Review

Before delivery, verify that the PRD:

- is source-backed and includes reference markers;
- begins with an honest revision table;
- contains no hidden chain-of-thought, drafting notes, or process narration;
- is concise, readable, and useful for human decision-making;
- includes measurable success metrics, testable acceptance criteria, and explicit non-goals when relevant;
- ends with a reference list containing every cited source.
