# PRD Writing Rules

## Hard Rules

- Write the PRD in professional, formal Simplified Chinese.
- Write for human readers; use clear, natural, reader-oriented prose.
- Put the revision table first in the document.
- Fill revision history honestly for every update: date, version, author, change summary, evidence or approval basis.
- Do not include process narration, self-description, hidden reasoning, "I researched", "next I will", or draft notes.
- Do not invent requirements, metrics, users, technical constraints, rollout plans, dependencies, or references.
- Mark all material statements with source IDs such as `[BIZ-01]`, `[USER-03]`, `[TECH-02]`, or methodology IDs such as `[M01]`.
- End the document with "参考文献" and list every source used.
- Keep content tightly bound to the PRD purpose. Remove unrelated background, generic product theory, and ornamental text.
- Prefer brevity. If a section cannot be source-backed, state the gap succinctly instead of padding.

## Format Rules

- Use Markdown headings with clear hierarchy.
- Use tables for revision history, stakeholder matrix, requirement inventory, scope boundaries, acceptance criteria, metrics, rollout plan, risk register, and traceability.
- Use Mermaid for user flow, system flow, state transition, or dependency diagrams when a visual improves clarity.
- Use LaTeX only for formulas that clarify measurement or prioritization.
- Use bullet lists for short parallel items; use prose for decisions and rationale.
