## 产品目标

Scheduler 的目标是成为独立的调度控制器与状态裁判：它把已经存在的工作流路线推进为可执行、可恢复、可审计的作业状态流，但不拥有路线设计权，也不执行具体任务。它的核心价值是用确定性规则回答“何时可以派发、何时可以接受、何时必须阻断”，并以保守、机械、可追溯的方式保护路线状态。依据：`CORE-001`、`USER-001`、`REQ-001`、`REQ-008`、`SRC-001`、`SRC-002`。

| 集成边界 | Scheduler 是 daemon 程序，其他程序通过 API 调用；权威状态与租约由持久化存储承载 | `TECH-001`, `TECH-002`, `SRC-002` |

## 实现方案

Scheduler 应围绕固定循环实现：读取路线、节点状态、作业定义、权限和租约；先处理过期或失效执行权；再判断可调度节点；对满足条件的节点创建作业实例与执行租约；读取 Executor 返回的运行结果与证据；根据身份、租约、fencing token、证据充分性和状态机规则，将作业实例标记为 accepted、rejected、expired 或 blocked，并推进节点状态。没有运行中作业且没有可调度节点时，只输出 route_exhausted，不输出业务成功。依据：`FLOW-001`、`MOD-001`、`MOD-002`、`MOD-003`、`STATE-001`、`STATE-002`、`EXE-001`、`EXE-003`。

## 验收标准与方法

验收必须使用 `spec-scheduler` 仓库内真实路线、节点、权限、租约和 Executor 结果样例，不使用 mock、stub 或纯口头验收。关键标准包括：重复派发不能产生重复 active job；过期租约或低 fencing token 结果必须被拒绝；缺少必要证据的 completed 结果不能使节点 deliveried；所有状态变化必须有事件、原因和证据引用；路线无运行作业且无可调度节点时必须输出 route_exhausted 且不宣布业务完成。依据：`AC-002`、`AC-003`、`AC-004`、`AC-005`、`AC-007`、`AC-008`、`VER-002`、`VER-003`、`VER-004`、`VER-005`、`VER-008`。

## 路线图

第一阶段交付 Scheduler 的产品需求与概要设计，覆盖 daemon API、持久化状态、调度循环、租约/fencing、结果证据校验、状态机和真实验收方案。后续阶段可以在 HLD 之后进入详细设计与实现，但不得由本工作流输出任务拆解或任务计划。依据：`PHASE-001`、`SCOPE-001`、`BAR-002`、`DONE-001`、`DONE-002`、`DONE-003`。

## 风险与待确认

主要风险是并发调度导致重复作业、陈旧 Executor 结果覆盖新状态，以及证据语义不清导致误把 completed 当作 deliveried。当前已确认 daemon API、持久化状态和仓库内真实用例三项关键边界；HLD 阶段仍需把 API 形态、事务边界、数据对象字段和验收命令设计到可实现级别。依据：`RISK-001`、`RISK-002`、`Q-001`、`Q-002`、`Q-003`、`TECH-002`、`TECH-003`。
